# Discovery Summary

> Phase 1.3 deliverable per [Feature-Dev SKILL.md](../../agents/Feature-Dev/SKILL.md).
> User-confirmed before architecture work continues.

| Field                                 | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Problem**                           | Data stewards spend hours per week manually classifying tables, tracing lineage impact before schema changes, and translating between OpenMetadata and the rest of their toolchain (GitHub, Slack). The OpenMetadata UI and REST API solve discovery, but the _governance loop_ — scan → classify → confirm → apply → notify — is still manual.                                                                                                                                                                                           |
| **Current Solution**                  | Stewards open the OM UI tab-by-tab, hand-tag PII columns one at a time, mentally simulate downstream impact from the lineage graph, then file GitHub issues by hand for each affected table owner.                                                                                                                                                                                                                                                                                                                                        |
| **V1 Success Metric**                 | A judge can clone the repo, run `make demo`, type `auto-classify PII in customer_db` in the chat UI, and see 50 tables scanned + tagged with PII labels (>85% LLM-tag agreement vs human spot-check on 5 sampled tables) in **<60 seconds at <$0.05 spend**, with every write gated by a human-in-the-loop confirmation.                                                                                                                                                                                                                  |
| **Must-Have Features (rank-ordered)** | 1. NL metadata search (calls `search_metadata` + `semantic_search`)<br>2. Auto-classification with human confirmation (calls `search_metadata` → `get_entity_details` → LLM → `patch_entity`)<br>3. Lineage impact analysis in plain English (calls `get_entity_lineage` + `root_cause_analysis`)<br>4. Multi-MCP cross-platform: OM PII finding → GitHub issue per table (Phase 3)                                                                                                                                                       |
| **Out of Scope (v1)**                 | Modifying `openmetadata-mcp/` Java module; building new OM connectors; production HA / DR / multi-tenant; custom LLM training; replacing OM UI; Slack/Teams integrations beyond the GitHub MCP demo                                                                                                                                                                                                                                                                                                                                       |
| **Platform**                          | Web (browser-served React UI) + local FastAPI backend — runs on the team member's laptop or a single Docker host                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Tech Constraints**                  | Must use the official `data-ai-sdk` (judges score "Best Use of OpenMetadata"); LLM is OpenAI GPT-4o-mini (free Codex Hackathon credits, zero cost blocker); Python 3.11+; Apache 2.0 license to match OM upstream                                                                                                                                                                                                                                                                                                                         |
| **Scale**                             | Demo: 1 user (judge), ~50 entities. 12-month: 10 concurrent users, 10K entities — but explicitly out-of-scope for v1; we document the path, we don't build it.                                                                                                                                                                                                                                                                                                                                                                            |
| **Timeline**                          | 7 days remaining (Apr 19 → Apr 26, 2026). Hard stop at Apr 26 hackathon submission deadline.                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **Sensitive Data**                    | **YES** — the agent processes table column names + descriptions which may contain PII identifiers. The agent itself does NOT handle real PII values, only metadata. But: (a) catalog content is attacker-controlled (any catalog editor can poison a description); (b) the agent's write tools (`patch_entity`, `create_glossary*`, `create_lineage`, GitHub `create_issue`) can mutate the catalog or external systems. Security architecture is **mandatory**, not optional. See [Security/ThreatModel.md](../Security/ThreatModel.md). |
| **Availability Target**               | Best-effort for v1 (single laptop, no SLA). Demo-day target: agent backend up for the full 5-min demo window with zero unhandled exceptions.                                                                                                                                                                                                                                                                                                                                                                                              |
| **Regulatory**                        | None claimed for v1 (Apache 2.0 OSS hackathon submission). PII tag suggestions are advisory; no GDPR/HIPAA/PCI claims made.                                                                                                                                                                                                                                                                                                                                                                                                               |
